## 你可能遇過這個問題

你裝好了 Claude Code，它可以幫你寫東西、改檔案、甚至做網頁。但你很快就發現：

**「它只看得到我電腦裡的檔案，看不到我的 Email、行事曆、Notion⋯⋯」**

沒錯。Claude Code 預設只能操作你電腦上的檔案。要讓它連上外部工具，大家第一反應是裝 **MCP**（Model Context Protocol）。但雷蒙想先幫你破除一個迷思：

**MCP 不是唯一的路，也不一定是最好的路。**

---

## 先看這個：MCP / API / CLI — 該用哪條路線？

雷蒙自己接了快一年的工具，歸納出這個決策順序：

| 優先度 | 路線 | 什麼時候用 | 優點 | 缺點 |
|:--:|:--|:--|:--|:--|
| 🥇 **先試** | **CLI 工具**（例如 `gh`、`gws-cli`、`notion-cli`） | 該服務有官方或穩定的命令列工具 | 不占 context、每次想用就跑 `gh api xxx`、行為可預期 | 需要學幾個指令（但 AI 會幫你記）|
| 🥈 **再試** | **REST API + .env**（curl / Python `requests`） | CLI 沒有，但該服務有穩定 REST API | 精準控制、可以複雜組合、錯誤訊息清楚、一樣不占 context | 要自己管 API key |
| 🥉 **最後才裝** | **MCP** | CLI + API 都沒有，或想要「AI 自己會用」的體驗 | 真的做到「AI 自動懂」 | **每次對話都會載入 MCP 定義，吃 context 空間**；MCP server 斷線會整個對話卡住 |

**為什麼雷蒙這樣排？**

- CLI/API 是「你喊一次才跑一次」，不用的時候完全不占 context
- MCP 是「每次對話一開始，Claude Code 就要把所有 MCP 的工具定義塞進系統 prompt」，你裝越多 MCP，對話開頭就越肥、越慢、越容易把其他 context 擠掉
- 而且 MCP server 跑在本地 process，有時候會斷、有時候會 hang，一斷整個對話就卡住

**雷蒙自己的組合**（給你當參考）：

- ✅ **CLI**：`gh`（GitHub）、`gws-cli`（Google Workspace 全套）、`zeabur`（Zeabur 部署）
- ✅ **REST API + .env**：WordPress（`curl -u ...`）、Notion（`curl` 打 `api.notion.com`）、Kit 電子報、Cloudflare、R2
- 🥉 **MCP**（少用）：Firecrawl（用它抓網頁最順）、Chrome DevTools（debug 前端用）

**有個反例**：很多人看到 Claude Code 生態系有 Notion MCP 就一定要裝。雷蒙實際測下來，**用 REST API 直接打 `api.notion.com` 更穩、更快、更可控**，Notion MCP 對複雜的資料庫操作常常會卡住。所以雷蒙的 notion-api skill 明確寫：「讀取用 MCP、寫入用 REST API」。

> [!IMPORTANT]
> **這份秘笈會依序帶你跑三條路線**
> 1. 先訪談你用什麼工具、做什麼事
> 2. 針對每個工具，**優先建議 CLI / API**，最後才考慮 MCP
> 3. CLI 的幫你確認已安裝、教 AI 基本指令
> 4. API 的幫你把 key 存到 `.env`，並寫一份 skill 當「這個 API 怎麼用」的參考
> 5. 真的只能用 MCP 的，才走下面 Section B 的 MCP 安裝流程

## 原本你可能遇過的問題（續）

回到大家最常想到的 MCP，即使你決定要裝 MCP，也會卡在：MCP 工具有幾百個，你根本不知道該裝哪些、怎麼裝、裝了之後怎麼用。

## 這份武功秘笈幫你做什麼

AI 會先訪談你：你平常用什麼工具、拿來做什麼。然後根據你的回答，**照 CLI → API → MCP 的優先順序**幫你做最適合的整合，不多裝一個 MCP，也不漏掉任何一個服務。

裝完之後你會得到：
- **一份你的工具連接組合**（CLI / API / MCP 混搭，每個工具用最省 context 的方式接）
- **`.env` 檔**（把 API key 集中管理，AI 需要時從 `.env` 讀，不會把 key 塞進 `CLAUDE.md`）
- **一份「你的 AI 現在會什麼」清單**（列出 AI 可以幫你做的具體事情，以及每件事走的是哪條路線）
- **踩坑排錯指南**（常見問題和解法，遇到直接查）
- **一份「我的工具清單」文件**（Section E，跟同學交換的作業材料）

## 雷蒙目前用了哪些工具？分別用哪條路線接？

給你一個參考，以下是雷蒙自己每天在用的工具組合，**注意看路線欄**：

| 工具 | 路線 | AI 能幫你做什麼 | 使用場景 |
|:--|:--|:--|:--|
| **Gmail** | 🥈 API（`gws-cli` 或 REST）| 掃收件匣、讀信、建草稿、搜尋舊信 | 「幫我看今天有什麼重要的信」 |
| **Google Calendar** | 🥇 CLI（`gws-cli calendar`）| 查行程、建活動、找空檔、搬時間 | 「下週三下午有空嗎？幫我排一個會議」 |
| **Notion** | 🥈 API（`curl api.notion.com`）| 查資料庫、建頁面、更新狀態 | 「幫我在任務庫新增一個待辦」 |
| **WordPress** | 🥈 API（`curl` + app password）| 查文章、建草稿、上傳媒體 | 「幫我把這篇存成 WP 草稿」 |
| **Kit 電子報** | 🥈 API（REST）| 建廣播、管訂閱者、查開信率 | 「把這篇推成 Kit 草稿」 |
| **GitHub** | 🥇 CLI（`gh`）| 管理 repo、看 issues、建 PR | 「幫我看這個 repo 最近的更新」 |
| **Zeabur** | 🥇 CLI（`zeabur`）+ skill | 部署、查 log、管環境變數 | 「幫我把這個專案部署上去」 |
| **Firecrawl**（抓網頁） | 🥉 MCP | 抓網頁內容、搜尋網路資料 | 「幫我整理這個網頁的重點」 |
| **Chrome DevTools**（debug 前端） | 🥉 MCP | 看頁面結構、截圖、讀 console | 「為什麼這個按鈕點下去沒反應？」 |

**觀察**：雷蒙接了 9 個服務，只有 2 個走 MCP，其餘全部走 CLI 或 API。這樣做的好處是**對話開頭的 context 負擔很輕**：只有真的需要 MCP 的那 2 個服務才會佔空間。

你不需要全裝。AI 會根據你的回答，只推薦你真正需要的。

## 怎麼做？

把這份文件丟給一個 Claude Code 對話，跟它說：

> 幫我跑「外部工具整合包」

AI 會依序：問你用什麼工具 → 確認要裝哪些 → **寫成一份整合計畫文件放進 `100_Todo/`** → 建立一份「AI 能幹清單」骨架 → 告訴你之後怎麼一次挑一個工具執行。全程約 15 分鐘。

**雷蒙的設計哲學**：這份秘笈**不會**一次把所有工具都裝完。為什麼？因為 MCP / API 的套件一直在更新，寫死的步驟半年後就過時了。所以秘笈做的是「寫計畫」，執行計畫的時候 AI 會**即時上網查當下最推薦的整合方式**，永遠跟著最新版本走。這才是真正可攜、不會被綁架的做法。

## 安裝指令（AI 執行區）

> **給 Claude Code 的總覽**：這份文件是「**外部工具整合包 by 雷小蒙**」的執行劇本。你要幫用戶把常用工具用最省 context 的方式整合到 Claude Code。請**按順序**執行 Section A → B → C → D → E，每一節完成後再進下一節。全程用繁體中文、親切口氣，避免術語。
>
> **四個絕對不能破壞的原則**：
> 1. **Section A 的訪談必須用 `AskUserQuestion` 工具跳出選項框**，不要用純文字讓用戶打字回答。
> 2. **每個服務優先走 CLI → API → MCP 的順序**。只有當前兩條路線都不通時才裝 MCP。這是這份秘笈的核心價值，避免 context 被無謂的 MCP 定義塞爆。
> 3. **只裝用戶選擇的工具**，不要自作主張多裝其他 MCP。
> 4. **每裝完一個整合，都要立即驗證它能正常運作**，不能跳過驗證就進下一個。

> **⚠️ 這份文件的 Section B 目前只有 MCP 安裝流程**（保留自 v1.x），CLI 和 API 兩條路線的詳細步驟還在補充中。當你執行到 Section B 時：
> - **CLI 工具**：直接告訴用戶「我建議你用 `gh` / `gws-cli` 接這個服務」，引導用戶安裝該 CLI 工具（用 brew / npm / pip），裝完跑一個驗證指令（例如 `gh auth status`），然後把常用 10 個指令寫進該服務的 skill。
> - **REST API**：幫用戶把 API key 存到 `600_Project/.env`（或 `~/.env.claude-tools`），在 `000_Agent/skills/` 底下建一個該服務的 skill（例如 `notion-api/SKILL.md`），skill 內容寫「怎麼用 curl 打這個 API 的基本操作、認證方式、常見 endpoint」。
> - **MCP**：按下方 Section B 流程安裝。
>
> 若 CLI/API 路線需要更完整的安裝劇本，告訴用戶「這部分的完整劇本還在雷蒙的 roadmap 上，目前先用這個快速版」，並把 TODO 寫進 daily log 提醒雷蒙補完。

## Section B：把訪談結果變成「整合計畫文件」，之後慢慢執行

> **這一節的設計哲學**（先讀這段）：
>
> 雷蒙原本想把所有工具的安裝步驟都寫死在這份文件裡（Gmail 要怎麼拿 OAuth、Notion 要怎麼建 Integration、Firecrawl 要去哪註冊⋯），但後來發現，**這樣寫死反而害了你**。
>
> 原因有三個：
>
> 1. **MCP / API / CLI 的安裝方式一直在變**。今天的 `@modelcontextprotocol/gmail-mcp` 下個月可能改名、換 npm 套件名、改用新的 auth flow。把步驟寫死在文件裡，半年後全部過時。
> 2. **Claude 4.6 / Opus 已經夠聰明**。它有網路搜尋能力，執行到某個工具時它會自己去查「這個工具現在最推薦的接法是什麼」、「官方 GitHub repo 的 README 怎麼寫」、「有沒有人剛好寫過踩坑文」。它查到的永遠比雷蒙寫死的新。
> 3. **你的工具組合非常個人化**。你訪談後的清單可能有雷蒙沒用過的工具，雷蒙寫死的步驟根本就不涵蓋。
>
> **所以 Section B 不再是「安裝劇本」，而是「計畫產生器」**：
>
> 1. 根據 A-6 的訪談結果，AI 幫你寫一份「整合計畫文件」放進 `100_Todo/integrations/`
> 2. 裡面每個工具一個區塊，有路線建議（CLI/API/MCP/瀏覽器）、目前狀態、具體 TODO checklist
> 3. **先不動手安裝**。計畫寫完後，這一輪 pro-kit 算跑完了
> 4. 之後你有空時，打開計畫文件跟 AI 說：「我們挑 [某個工具] 來裝」，AI 會**即時上網查**該工具當下最新的整合方式，一步一步帶你裝完、驗證、打勾
>
> **為什麼這樣比較好**：
> - 計畫跟執行分開，你不會在一個 session 裡被「裝 6 個工具」搞到精神耗盡
> - 每次執行都用最新資訊，不會踩「文件寫的步驟已經過時」的坑
> - 計畫文件留在 `100_Todo/` 是你可查可改的真實工作清單，不是一次性的秘笈
> - **學到可攜性的實作**：同一份計畫文件，未來換 AI 大腦時還是看得懂、照樣能繼續執行

## 決策原則速查

在選每個工具的路線前，優先順序是：

1. 🥇 **CLI**（`gh`、`gws-cli`、官方 CLI）— 不吃 context、最穩定
2. 🥈 **REST API + `.env`**（curl / Python requests）— 彈性最高、可精準控制
3. 🥉 **MCP**（`~/.claude.json` 的 `mcpServers`）— 只有 CLI + API 都不行時才用
4. 🔒 **瀏覽器控制**（Chrome DevTools MCP / Playwright）— 真的沒 API 才走這條

每個工具的「建議路線」欄位是 AI 初步判斷，實際執行時會再用網路搜尋確認當下最新的最佳做法。

3. **不要**在計畫文件裡寫具體 npm 套件名、具體設定路徑、具體 auth URL，那些會過時。**只寫原則 + 要查的問題**，實際安裝步驟等執行時 AI 自己查
4. 計畫文件寫完後**直接 Read 一次驗證**，確認 frontmatter 完整、每個工具都有區塊

### B-2. 告訴用戶「計畫寫完了，執行看你什麼時候有空」

用下面這段話（原封不動）告訴用戶：

> 📋 **你的整合計畫寫好了！位置：`100_Todo/integrations/[YYYY-MM-DD]-tool-integration-plan.md`**
>
> 裡面有 [N] 個工具要整合，每個都有建議路線和要查的事情。
>
> **接下來不用急著全裝**。雷蒙的建議是：
>
> 1. **先打開計畫看一眼**（3 分鐘），挑一個「你每天一定會用到」的工具當第一個
> 2. **今天或明天有空時**，打開一個新的 Claude Code 對話，說：
>
>    > 幫我從 `100_Todo/integrations/[YYYY-MM-DD]-tool-integration-plan.md` 挑 [工具名] 來裝。
>    > 請先用網路搜尋查這個工具現在官方推薦的 AI agent 整合方式，然後再動手。
>    > 裝完把 checklist 打勾、在備註欄寫下實際用的方案。
>
> 3. **AI 會做的事**：web search 查最新資訊 → 告訴你它查到什麼 → 跟你確認路線 → 動手裝 → 實測一個指令驗證 → 更新計畫文件
> 4. **一週一個工具**就好。裝完一個用熟了再裝下一個，不要貪快一次裝 5 個
>
> **為什麼要這樣拖時間？**
>
> 因為 MCP / API / CLI 的套件一直在更新，今天寫死的步驟半年後可能就過時了。分散執行、每次都讓 AI 現查現裝，**你永遠跟著最新版本走**，不會被老舊的教學文綁架。這個思路也對齊 2-4「可攜性」的哲學：你的計畫文件不依賴任何特定版本，是跟著你走的。

### B-3. 「執行模式」的具體劇本（給未來的 AI session 看）

這段**寫在計畫文件的最底部**，當未來某個 AI session 被叫去執行某個工具時，它會讀到這段指引。內容如下：

```markdown
---

## 給未來 AI 執行時的指引（不要刪這段）

當用戶打開這份文件跟你說「幫我挑 [某個工具] 來裝」時，請按以下步驟：

### 1. 確認範圍

用 `AskUserQuestion` 確認：
- 你要整合 [工具名]，對嗎？
- 整合的主要用途是什麼？（從計畫文件的「用途」欄讀出來讓他確認）

### 2. 用網路搜尋查最新整合方式

**這一步絕對不要跳過，也不要用你訓練資料裡的舊資訊。** 執行：

1. 用 WebSearch / WebFetch 或 MCP 的搜尋工具查以下問題：
   - `"[工具名]" Claude Code MCP integration 2026`
   - `"[工具名]" official CLI tool`
   - `"[工具名]" REST API authentication`
2. 優先看官方文件、GitHub README、官方 blog 公告
3. 對照計畫文件的「建議路線」，看看有沒有更新/更好的方案
4. **把你查到的結果整理成一段話告訴用戶**：「我查到 [工具名] 目前最推薦的整合方式是 [XX]，因為 [原因]。要不要照這個走？」
5. 用 `AskUserQuestion` 讓用戶拍板

### 3. 執行安裝

根據拍板的路線：

- **CLI 路線**：幫用戶安裝該 CLI 工具（brew / npm / pip，依你查到的官方推薦），引導完成 auth，跑一個驗證指令
- **API 路線**：
  - 引導用戶取得 API key / token（告訴他官方取得頁面的網址）
  - 存到 `600_Project/.env` 或 `~/.env.claude-tools`，key 命名用大寫 + 底線（例如 `NOTION_TOKEN`）
  - 在 `000_Agent/skills/` 底下建一個該服務的 skill（例如 `000_Agent/skills/notion-api/SKILL.md`），寫「怎麼用 curl 打這個 API 的認證、常見 endpoint、範例」
- **MCP 路線**：
  - 編輯 `~/.claude.json` 的 `mcpServers` 區段加入新的 entry
  - 完成 auth（OAuth 跳瀏覽器、或貼 token）
  - 重新載入 MCP（告訴用戶怎麼 restart 或讓 Claude Code 重新掃描）
- **瀏覽器控制**（最後手段）：走 Chrome DevTools MCP 或 Playwright，但要先警告用戶這條路最不穩

### 4. 驗證

用一個實際指令測試這個整合真的能用。例如：
- Gmail → 「幫我看最近 3 封信的主旨」
- Calendar → 「今天的行程」
- Notion → 「搜尋 Notion 裡有 `AI 共學島` 的頁面」
- GitHub → 「列出我的 repos」

### 5. 更新計畫文件

用 `Edit` 工具更新 `[這份計畫檔名]`：
- 該工具區塊的標題從 🟡 改成 🟢
- 安裝 checklist 全部打勾
- 備註欄寫：「實際用了 [路線]、裝的套件版本 [版本號]、踩坑 [...]、驗證指令 [...]」
- 進度總覽數字調整

### 6. 告訴用戶下一步

「[工具名] 整合完成！剩下 [N] 個工具。建議一週後再挑下一個來裝，讓這個先用熟。」
```

## Section D：踩坑排錯速查

> 把以下常見問題告訴用戶，並存到用戶可以查閱的地方。

### 常見問題

**Q：安裝後重開 Claude Code，MCP 就不見了？**
> MCP 設定存在 `~/.claude.json`，重開不會消失。如果真的不見了，用 Read 工具檢查 `~/.claude.json` 裡面的 `mcpServers` 區段是否還在。

**Q：Gmail 突然不能用了，說授權過期？**
> Google OAuth Token 有時效性。重新執行一次授權流程即可。

**Q：Notion 搜尋不到某些頁面？**
> Notion Integration 需要被「邀請」到對應的頁面或資料庫。到 Notion 裡面，打開那個頁面 → 右上角「⋯」→「Connect to」→ 選擇你的 Integration。

**Q：Firecrawl 說額度用完了？**
> 免費方案有每月額度限制。可以升級付費方案，或改用 WebFetch（Claude Code 內建，免費但功能較少）。

**Q：裝了 MCP 但 AI 好像沒在用？**
> 有時候 AI 不知道它有這些工具。直接跟它說：「你現在有 Gmail MCP，幫我看信。」明確告訴它用哪個工具。

## Section E：產出「個人工具清單」文件 + 作業引導（新增）

> 這一節是**整份秘笈最有長期價值的部分**。MCP 安裝完只是第一步，真正讓 AI 分身「懂你怎麼工作」的是一份**完整盤點你工具組合的清單**。雷蒙自己有一份 `000_Agent/knowledge/雷蒙數位工具清單.md`，AI 在遇到「推薦工具」「評估替代方案」「寫工具比較文」時都會翻它，這就是 Section E 要讓學員做的東西。

### E-1. 先生成起手式清單

根據 Section A 訪談 + Section B 安裝結果，**自動產出**一份 `000_Agent/knowledge/我的工具清單.md`（如果用戶沒跑過 pro-kit 01，就改放在 `~/my-tool-inventory.md`）。

### E-2. 告訴用戶這份清單會被 AI 怎麼用

用下面這段話（原封不動）告訴用戶：

> - 「我想找一個新的專案管理工具，推薦幾個？」→ AI 會知道你已經用 Notion + Linear，不會又推薦這兩個
> - 「幫我寫一篇部落格比較 A 工具 vs B 工具」→ AI 會用你自己的評估邏輯寫，不是抄網路上的廢話文
> - 「我這個工作流有沒有可以優化的地方？」→ AI 會看懂你現在的串接，給出具體建議
> - 「這個新工具值得試嗎？」→ AI 會對照你淘汰的工具，判斷你真正的痛點
>
> **關鍵**：這份清單的「已淘汰工具」欄位其實最值錢。那是你花錢踩過的坑。AI 看到就知道「這條路你已經走過不用再推薦」。
